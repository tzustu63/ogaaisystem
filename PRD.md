# 產品需求文件 (PRD)
## 高等教育國際化策略執行管理系統

---

## 一、產品概述

### 1.1 產品定位
本系統為高等教育機構打造的三層式策略執行管理平台，整合平衡計分卡(BSC)、目標與關鍵結果(OKR)、責任分配矩陣(RACI)、看板管理(Kanban)及PDCA循環，實現從教育部政策到日常執行的完整追蹤與治理。

### 1.2 核心價值主張
- **策略對齊**：確保每項任務都能追溯至校級戰略目標
- **權責明確**：透過RACI與工作流引擎消除跨單位灰區
- **數據治理**：建立單一真相來源，支援稽核與視導需求
- **持續改善**：結構化PDCA循環，將改善行動轉化為可追蹤證據

### 1.3 目標用戶
- **主要用戶**：國際處處長/副校長（戰略層）、各組組長（戰術層）、專員（執行層）
- **協作用戶**：跨單位顧問（C角色）、知會對象（I角色）
- **外部相關方**：教育部視導、國際排名機構、稽核單位

---

## 二、系統架構：三層式設計

### 2.1 第一層：戰略層 (Strategy Layer)
**目標**：將教育部政策與校級願景轉化為可衡量的平衡計分卡

### 2.2 第二層：戰術層 (Tactical Layer)
**目標**：用OKR拆解戰略，用RACI明確跨單位權責

### 2.3 第三層：執行層 (Execution Layer)
**目標**：用Kanban落地任務，用PDCA驅動持續改善

### 2.4 核心設計原則
> **關鍵要求**：三層之間的連結必須為系統強制規則（資料結構+工作流），避免淪為「做一堆表、最後靠人追」的傳統模式

---

## 三、核心模組需求

## 【模組1：KPI Registry - 指標字典與治理中樞】

### 3.1 模組定位
作為全系統的「單一真相來源」，解決以下痛點：
- 同一KPI不同人算法不同
- 閾值各自定義導致燈號失真
- 稽核要求證明數據依據時無法回溯

### 3.2 功能需求

#### 3.2.1 KPI主檔管理
每個KPI必須包含以下欄位（**強制必填**）：

| 欄位名稱 | 說明 | 範例 |
|---------|------|------|
| KPI_ID | 唯一識別碼 | BSC-F-001 |
| KPI名稱 | 中英文名稱 | 外部經費成長率 |
| 所屬BSC構面 | 財務/客戶/內部流程/學習成長 | 財務構面 |
| 定義 (Definition) | 包含/不包含範圍 | 含教育部深耕、雙語計畫，不含產學合作 |
| 公式 (Formula) | 明確到欄位級 | (本年度外部經費總額 - 去年度外部經費總額) / 去年度外部經費總額 × 100% |
| 資料來源 (Source) | 系統/表單/人工/API | 主計系統API + 計畫管理系統 |
| 資料負責人 (Data Steward) | 負責數據品質的人員 | 研發處主計組_張組長 |
| 更新頻率 | Monthly/Quarterly/Ad-hoc | Quarterly |
| 目標值 (Target) | 年度/季度目標 | 年度: 15%, Q1: 3%, Q2: 7%... |
| 閾值 (Thresholds) | 紅黃綠規則 | 綠: ≥90%目標, 黃: 70-89%, 紅: <70% |
| 證據要求 | 需上傳的佐證資料 | 核銷清單、經費核定公文、執行率報表 |
| 適用計畫標籤 | 多選 | 深耕、雙語(BEST)、新南向、學海 |

#### 3.2.2 閾值模式設定
支援三種閾值模式（可設定模板）：

**模式1：固定值**
```
範例：簽證合規率
- 綠燈：≥ 98%
- 黃燈：95-97.9%
- 紅燈：< 95%
```

**模式2：相對值（同比/環比）**
```
範例：境外生招生人數
- 綠燈：≥ 去年同期 110%
- 黃燈：去年同期 90-109%
- 紅燈：< 去年同期 90%
```

**模式3：預測偏離（進階功能）**
```
範例：季度專案執行率
- 綠燈：照目前速度季末達標機率 ≥ 80%
- 黃燈：達標機率 60-79%
- 紅燈：達標機率 < 60%
```

#### 3.2.3 版本控管
- 每次修改KPI定義/公式/閾值時，系統自動創建新版本
- 保留完整變更歷程（誰在何時改了什麼）
- 歷史數據自動關聯對應版本的KPI定義

## 【模組2：戰略地圖與儀表板 (Strategy Map & Dashboard)】

### 3.3 戰略地圖 (Strategy Map)

#### 3.3.1 因果鏈建立
- 支援「目標→目標」的因果連線繪製
- 視覺化呈現四構面之間的邏輯關係

**因果鏈範例**：
```
學習成長：行政人員雙語能力提升
    ↓
內部流程：雙語行政流程效率提高
    ↓
客戶：境外生滿意度提升、留存率提升
    ↓
財務：外部經費與自償收入增加
```

#### 3.3.2 BSC四構面目標管理
- 財務構面 (Financial)
- 客戶構面 (Customer)
- 內部流程構面 (Internal Process)
- 學習與成長構面 (Learning & Growth)

每個目標必須關聯：
- 1個或多個KPI
- 對應的Initiatives（策略專案）
- 責任人與協作單位

### 3.4 戰略儀表板 (Strategy Dashboard)

#### 3.4.1 四種視圖模式

**視圖1：年度達成率總覽**
- 四構面達成率雷達圖
- 各KPI達成率橫條圖
- 交通燈號統計（紅/黃/綠數量）

**視圖2：趨勢圖 (Trend)**
- 近12期/24期走勢圖
- 支援同比/環比對照
- 目標線與實際值雙軸顯示

**視圖3：領先/落後指標標記**
- 標記Leading Indicators（可提前干預）
- 標記Lagging Indicators（結果指標）
- 依權重排序關鍵指標

**視圖4：下鑽路徑 (Drill-down)**
```
KPI → 對應OKR → 對應Initiative → 對應任務 → 證據附件
```
點擊任一層級可進入詳情頁，支援層層追溯

#### 3.4.2 交通燈 (Traffic Lights) 顯示規則
- 自動依KPI Registry中的閾值設定計算燈號
- 支援手動標記例外情況（需填寫說明）
- 燈號變更時自動通知相關責任人

## 【模組3：策略專案與OKR管理 (Initiatives & OKR)】

### 3.5 策略專案 (Initiative / Program)

#### 3.5.1 設計理念
解決「KPI很高層，Kanban很瑣碎」的中間斷層問題：
- **KPI** = 結果（What we achieve）
- **Initiative** = 做法（What we do）
- **OKR** = 分段里程碑（How we measure progress）
- **Task** = 每天要做的事（Daily execution）

#### 3.5.2 Initiative主檔欄位

| 欄位 | 說明 | 必填 |
|-----|------|-----|
| Initiative ID | 專案編號 | ✓ |
| 專案名稱 | 中英文 | ✓ |
| 關聯BSC目標 | 多選 | ✓ |
| 影響的KPI | 多選，標記預期影響方向 | ✓ |
| 專案類型 | 政策回應/排名提升/風險管控/創新實驗 | ✓ |
| 負責單位 | 主責單位 | ✓ |
| RACI矩陣 | 必須指定至少1個A | ✓ |
| 起訖日期 | 開始日、結束日 | ✓ |
| 預算 | 若涉及經費 | - |
| 狀態 | 規劃中/執行中/已完成/已取消 | ✓ |
| 風險標記 | 高/中/低 | ✓ |

#### 3.5.3 與教育部視導的對應
- 每個Initiative可標記「適用計畫」（深耕/雙語/新南向...）
- 一鍵產生「該計畫下所有Initiatives清單」供視導使用
- 自動彙整該Initiative下的所有執行證據

### 3.6 OKR管理

#### 3.6.1 OKR結構
每個Initiative可設定季度OKR：
- **Objective (O)**：定性描述，例如「建立境外生完整生活支援體系」
- **Key Results (KR)**：3-5個可量化結果，例如：
  - KR1: 24小時緊急通報回應率達100%
  - KR2: 境外生滿意度調查≥4.5/5.0
  - KR3: 重大校安事件處置SLA達標率≥95%

#### 3.6.2 OKR與任務的關聯
- 每個KR可展開為多個Kanban任務
- 任務完成度自動回算KR進度
- KR達成時自動通知相關人員

## 【模組4：RACI權責矩陣與工作流引擎】

### 3.7 RACI模板管理

#### 3.7.1 場景模板預設
系統預設常見場景模板（可自訂新增）：

**模板範例：雙聯學制合約簽署**
```
R (Responsible): 國際合作組_承辦人
A (Accountable): 國際處處長
C (Consulted): 教務處註冊組、法規會、各系所主管
I (Informed): 副校長、主任秘書
```

#### 3.7.2 可執行工作流 (Executable Workflow)

每個RACI模板擴充為包含以下結構：

| 欄位 | 說明 |
|-----|------|
| 步驟 (Steps) | Draft → Consultation → Approval → Archive |
| 每步驟角色權限 | 定義R/C/A/I在該步驟的操作權限 |
| SLA（服務水平協議） | 例如：C需在5個工作天內會簽 |
| 必要附件 | 例如：學分對照表、法規檢核表 |
| 逾期升級 (Escalation) | 逾期自動通知上級（國際長/副校長） |
| 條件檢核 | A完成前必須滿足的條件 |

#### 3.7.3 工作流執行規則

**規則1：草案階段**
- R上傳草案後，系統自動通知所有C進行會簽
- C可在系統內標記「同意/建議修正/不同意」並附意見

**規則2：會簽階段**
- 顯示會簽進度儀表板（已完成/進行中/逾期）
- 逾期自動發送提醒郵件
- 達到SLA閾值時自動升級通知

**規則3：核定階段**
- A完成前系統檢查：
  - 所有C都已完成會簽
  - 必要附件已齊備
  - 若有「不同意」意見，需R回應處理
- A可「核定/退回修正」

**規則4：歸檔階段**
- 完成後自動歸檔至知識庫
- I角色收到完成通知
- 生成完整稽核軌跡報告

#### 3.7.4 稽核軌跡 (Audit Log)
記錄內容：
- 時間戳記
- 操作人員
- 操作類型（建立/修正/會簽/核定/退回）
- 修改前後內容對比
- IP位址（選配）

| 欄位分類 | 欄位名稱 | 說明 | 必填 |
|---------|---------|------|-----|

| | 任務描述 | 詳細說明 | - |

| **任務類型** | 類型 | 例行/專案/緊急事件 | ✓ |

- 待審核 (Review)

- 依單位分組

> **關鍵設計**：避免Kanban淪為孤立的待辦清單

- 專案類型任務：必須關聯至少1個Initiative或KR
- 若任務標記為「影響KPI」，完成時必須填寫「實際影響說明」
- 完成任務時，若有關聯KR，系統自動更新KR進度

## 【模組6：緊急事件管理 (Incident Management)】

### 3.9 Incident模組設計

#### 3.9.1 適用場景
- 境外生車禍/意外事故
- 簽證/居留問題緊急處理
- 突發公共衛生事件
- 校安通報事件

#### 3.9.2 Incident卡片欄位

| 欄位 | 說明 | 必填 |
|-----|------|-----|
| 事件編號 | 自動產生，例如INC-2025-001 | ✓ |
| 事件類型 | 校安/醫療/法務/簽證/其他 | ✓ |
| 嚴重程度 | Critical/High/Medium/Low | ✓ |
| 發生時間 | 時間戳記 | ✓ |
| 發生地點 | 校內/校外位置 | ✓ |
| 當事人 | 學生姓名、學號、聯絡方式 | ✓ |
| 緊急聯絡人 | 家長/監護人資訊 | ✓ |
| 事件描述 | 詳細經過 | ✓ |
| RACI | 必須指定A | ✓ |
| 處置清單 (Checklist) | 結構化待辦項 | ✓ |
| 通報紀錄 | 已通報單位與時間 | ✓ |
| 追蹤與結案報告 | Action taken & Prevention | - |

#### 3.9.3 處置清單 (Checklist) 範本
系統預設常見事件類型的Checklist範本：

**範例：境外生交通事故**
```
□ 確認傷勢並聯繫救護車（若需要）
□ 通知學生緊急聯絡人
□ 通報校安中心
□ 通報國際處
□ 陪同就醫並協助醫療翻譯
□ 申請保險理賠
□ 追蹤復原狀況
□ 評估是否需心理諮商
□ 完成事件報告
□ 檢討預防措施
```

#### 3.9.4 協作與權限
- Incident建立後，自動通知RACI中所有角色
- C角色（如校安、心理諮商）可在系統內協作並留下處置紀錄
- A角色確認所有Checklist完成後方可結案
- 結案必須填寫「預防復發措施」

## 【模組7：PDCA循環與改善追蹤】

### 3.10 PDCA四階段設計

#### 3.10.1 Plan（計畫）
- 關聯至Initiative或OKR
- 設定檢核點與頻率
- 指定負責人與數據來源

#### 3.10.2 Do（執行）
- 任務在Kanban中執行
- 證據附件上傳
- 進度自動回寫

#### 3.10.3 Check（檢核）- 核心強化

**功能1：自動回報排程**
- 系統依設定頻率（每週/每月/每季）自動抓取名單
- 發送檢核提醒給負責人
- 未回報時自動升級通知

**功能2：資料品質檢核 (Data Quality Check)**

每次回報時，系統自動執行三項DQ檢核：

| 檢核項目 | 說明 | 範例 |
|---------|------|------|
| **完整性 (Completeness)** | 該填的欄位是否缺漏 | 居留證到期日為空白 |
| **準時性 (Timeliness)** | 是否超過更新頻率 | 境外生名冊已3個月未更新 |
| **一致性 (Consistency)** | 同一實體在不同系統是否矛盾 | 學生A在學籍系統顯示在學，居留證系統顯示已離境 |

檢核結果：
- ✓ 通過：綠燈
- ⚠ 警告：黃燈，需人工確認
- ✗ 失敗：紅燈，阻擋該筆資料進入系統

**功能3：差異分析**
- 自動比對「計畫值 vs 實際值」
- 若偏離超過設定閾值，強制要求填寫原因

#### 3.10.4 Act（改善）- 結構化對策

**改善對策欄位設計**

| 欄位 | 類型 | 說明 |
|-----|------|------|
| Root Cause | 下拉選單 | 人力/流程/系統/外部政策/合作方 |
| 對策類型 | 下拉選單 | 補救(Corrective)/預防(Preventive) |
| 具體行動 (Action Items) | 文字+任務生成 | 可直接轉為Kanban任務 |
| 預期影響KPI | 多選 | 標記哪些KPI會因此改善 |
| 完成驗收方式 | 文字 | 如何驗證改善有效 |
| 負責人 | 人員 | 改善行動負責人 |
| 預計完成日 | 日期 | Deadline |

**改善追蹤儀表板**
- 顯示所有進行中的改善行動
- 依Root Cause分類統計（找出系統性問題）
- 追蹤改善行動對KPI的實際影響

## 【模組8：數據整合與來源管理】

### 3.11 分級整合策略

#### Level 1（MVP必備）：檔案匯入

**功能需求**：
- 支援CSV/Excel上傳
- 欄位映射介面（拖拉對應）
- 資料驗證規則（格式/必填/範圍）
- 匯入預覽與錯誤提示
- 版本控管（每次匯入保留歷史版本）

**匯入流程**：
```
上傳檔案 → 選擇對應資料表 → 欄位映射 → 資料驗證 → 
預覽 → 確認入庫 → 生成匯入報告
```

**版本控管**：
- 每次匯入記錄：時間、操作人、檔案名稱、筆數
- 支援版本回溯與對比
- 供稽核時查證數據來源

#### Level 2（第二階段）：校內系統對接

**建議對接優先順序**：

1. **學籍/SIS系統**
   - 境外生基本資料
   - 在學狀態（在學/休學/退學/畢業）
   - 選課與學分數
   
2. **財務/主計系統**
   - 計畫經費核定與執行
   - 核銷進度
   - 執行率計算

3. **人資/研習系統**
   - 行政人員基本資料
   - 培訓與研習紀錄
   - 語言證照資訊

**技術方案**：
- API串接（優先）
- 定時ETL（若無API）
- 資料同步頻率：即時/每日/每週

#### Level 3（進階）：外部資料與填報工作台

**QS/THE排名填報支援**：
- 建立填報指標清單
- 追蹤填報完整率
- 資料佐證檢核清單
- 截止日提醒

**學術產出（進階功能）**：
- 國際合著論文統計
- 可能資料源：Scopus API / Web of Science / 機構典藏

## 【模組9：權限管理與個資合規】

### 3.12 權限模型：RBAC + Scope

#### 3.12.1 角色定義 (RBAC)

| 角色 | 權限範圍 | 典型職位 |
|-----|---------|---------|
| 系統管理員 | 全系統設定、用戶管理 | 資訊中心 |
| 國際長/副校長 | 查看所有數據、核定權限 | 高階主管 |
| 處長/主任 | 查看單位數據、核定單位內事項 | 中階主管 |
| 組長 | 管理組內專案與任務 | 基層主管 |
| 專員 | 執行任務、上傳證據 | 承辦人 |
| 跨單位顧問 (C) | 會簽權限、查看相關資料 | 法規會、教務處 |
| 知會對象 (I) | 唯讀、接收通知 | 相關單位 |

#### 3.12.2 資料範圍 (Scope)

| Scope層級 | 說明 | 範例 |
|-----------|------|------|
| 全校 | 不限制 | 校長、國際長 |
| 學院 | 限定學院 | 工學院國際業務 |
| 系所 | 限定系所 | 資工系境外生導師 |
| 特定專案 | 限定Initiative | 雙聯學制專案成員 |
| 特定學生群組 | 限定名單 | 某屆交換生 |

#### 3.12.3 必備安全特性

**1. 身份驗證**
- SSO（Single Sign-On）校內帳號整合
- 選配：MFA（Multi-Factor Authentication）

**2. 欄位級遮罩**
敏感欄位依角色顯示：
- 護照號：顯示末四碼，例如 ****1234
- 居留證號：顯示末六碼
- 家長聯絡方式：需特定權限

**3. 下載與匯出控管**
- 名單匯出需額外權限
- 匯出記錄進入Audit Log
- 可設定「禁止匯出」欄位

**4. 完整稽核軌跡**
記錄內容：
- 誰查看了哪筆資料（View Log）
- 誰匯出了什麼資料（Export Log）
- 誰修改了什麼欄位（Edit Log）
- 時間戳記與IP位址

**5. 個資合規檢查清單**
- 資料蒐集目的告知
- 當事人同意書管理
- 資料保存期限管理
- 資料刪除請求處理（GDPR Right to be Forgotten）

## 六、成功關鍵因素 (CSF)

### 6.1 組織層面
1. **高階主管支持**：國際長/副校長需親自使用並推動
2. **跨單位協作**：資訊中心、教務處、主計室需配合
3. **變革管理**：提供完整教育訓練，避免抗拒

### 6.2 技術層面
1. **資料品質**：垃圾進垃圾出，需建立DQ規範
2. **系統整合**：與現有校務系統對接需投入資源
3. **效能優化**：大量數據時需考慮快取與索引

### 6.3 使用者層面
1. **簡化操作**：不要讓系統變成"更多表單"
2. **即時回饋**：操作後立即看到影響（例如KPI變化）
3. **價值感知**：讓用戶感受到"系統幫我省時間"而非"多一個工作"

## 結語

本系統的核心價值在於**將策略執行從「靠人追」轉變為「系統驅動」**。透過三層架構的緊密連結，每一個日常任務都能追溯到戰略目標，每一個KPI都有清晰的改善路徑。

成功的關鍵不在於功能有多炫，而在於：
1. **讓數據成為單一真相來源**（KPI Registry）
2. **讓權責清晰到無法推諉**（RACI + 工作流）
3. **讓改善變成系統性習慣**（PDCA + 結構化對策）

期待本系統成為高等教育國際化治理的標竿案例。
